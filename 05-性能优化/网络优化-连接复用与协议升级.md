# 网络优化-连接复用与协议升级

## 问题

> 网络优化你们做了哪些？连接复用怎么实现的？HTTP/2 和 QUIC 有什么区别？DNS 优化怎么做？

## 核心答案（30秒版本）

网络优化从连接层、协议层、数据层三个维度入手。连接复用通过 OkHttp 连接池实现（默认 5 个空闲连接，存活 5 分钟）。HTTP/2 多路复用解决了 HTTP/1.1 队头阻塞问题，但 TCP 层仍有队头阻塞；QUIC（HTTP/3）基于 UDP 彻底解决队头阻塞，支持 0-RTT 建连和连接迁移。DNS 优化通过 HTTPDNS 避免 LocalDNS 劫持和解析慢的问题。

## 深入解析

### 原理层（源码级）

**OkHttp 连接池源码：**

```kotlin
// okhttp3/internal/connection/RealConnectionPool.kt
class RealConnectionPool(
    maxIdleConnections: Int = 5,        // 最大空闲连接数
    keepAliveDuration: Long = 5,        // 存活时间
    keepAliveDurationUnit: TimeUnit = TimeUnit.MINUTES
) {
    private val connections = ConcurrentLinkedQueue<RealConnection>()

    // 获取可复用连接
    fun callAcquirePooledConnection(
        address: Address,
        call: RealCall,
        routes: List<Route>?,
        requireMultiplexed: Boolean
    ): Boolean {
        for (connection in connections) {
            // 检查连接是否可复用
            if (requireMultiplexed && !connection.isMultiplexed) continue
            if (!connection.isEligible(address, routes)) continue
            // 找到可复用连接
            call.acquireConnectionNoEvents(connection)
            return true
        }
        return false
    }

    // 连接清理（后台线程）
    fun cleanup(now: Long): Long {
        var longestIdleDuration = -1L
        var idleConnectionCount = 0

        for (connection in connections) {
            if (connection.calls.isNotEmpty()) continue // 使用中
            idleConnectionCount++

            val idleDuration = now - connection.idleAtNs
            if (idleDuration > longestIdleDuration) {
                longestIdleDuration = idleDuration
                longestIdleConnection = connection
            }
        }

        return when {
            longestIdleDuration >= keepAliveDurationNs -> {
                // 超时，移除
                connections.remove(longestIdleConnection)
                0 // 立即再次清理
            }
            idleConnectionCount > maxIdleConnections -> {
                // 超过最大空闲数，移除最久的
                connections.remove(longestIdleConnection)
                0
            }
            else -> keepAliveDurationNs - longestIdleDuration // 下次清理时间
        }
    }
}
```

**HTTP/1.1 vs HTTP/2 vs HTTP/3 对比：**

```
HTTP/1.1:
┌─────────────────────────────────────┐
│ TCP 连接 1: [请求A] → [响应A] → [请求B] → [响应B]  │ 串行（队头阻塞）
│ TCP 连接 2: [请求C] → [响应C]                      │ 浏览器限制 6 个并发
└─────────────────────────────────────┘

HTTP/2 多路复用:
┌─────────────────────────────────────┐
│ 单个 TCP 连接:                       │
│   Stream 1: [请求A帧][响应A帧]       │ 多个 Stream 交错传输
│   Stream 2: [请求B帧][响应B帧]       │ 应用层无队头阻塞
│   Stream 3: [请求C帧][响应C帧]       │ TCP 层仍有队头阻塞
└─────────────────────────────────────┘
问题：TCP 丢包时所有 Stream 都被阻塞

HTTP/3 (QUIC):
┌─────────────────────────────────────┐
│ 单个 UDP 连接:                       │
│   Stream 1: [请求A][响应A]           │ 独立 Stream，互不影响
│   Stream 2: [请求B][响应B]           │ Stream 2 丢包不影响 1 和 3
│   Stream 3: [请求C][响应C]           │ 彻底解决队头阻塞
└─────────────────────────────────────┘
额外优势：0-RTT 建连、连接迁移（WiFi→4G 不断连）
```

**QUIC 0-RTT 建连原理：**

```
首次连接（1-RTT）：
Client → Server: ClientHello + 支持的加密套件
Server → Client: ServerHello + 证书 + 加密参数
Client → Server: 验证完成 + 应用数据

后续连接（0-RTT）：
Client → Server: ClientHello + 缓存的加密参数 + 应用数据（立即发送！）
Server → Client: ServerHello + 响应数据

对比 TCP+TLS 1.3: 首次 2-RTT，后续 1-RTT
QUIC: 首次 1-RTT，后续 0-RTT
```

**HTTPDNS 原理：**

```
传统 DNS 解析链路：
App → LocalDNS(运营商) → 根DNS → 顶级DNS → 权威DNS
问题：LocalDNS 劫持、解析慢、缓存不可控、跨网调度不准

HTTPDNS：
App → HTTP(S) 请求 → HTTPDNS 服务器 → 直接返回 IP
优势：绕过 LocalDNS、防劫持、精准调度、可控缓存
```

```kotlin
    .connectionPool(ConnectionPool(

    .readTimeout(30, TimeUnit.SECONDS)

    // 连接失败重试
    .retryOnConnectionFailure(true)
    .build()
```

**2. HTTPDNS 实现：**

```kotlin
class HttpDns : Dns {
    private val cache = LruCache<String, List<InetAddress>>(100)
    private val httpDnsApi = "https://httpdns.your-service.com/resolve"

    override fun lookup(hostname: String): List<InetAddress> {
        // 1. 查本地缓存
        cache.get(hostname)?.let { return it }

        // 2. HTTPDNS 解析
        try {
            val response = httpClient.newCall(
                Request.Builder()
                    .url("$httpDnsApi?host=$hostname")
                    .build()
            ).execute()

            val ips = parseIps(response.body?.string())
            if (ips.isNotEmpty()) {
                cache.put(hostname, ips)
                return ips
            }
        } catch (e: Exception) {
            // HTTPDNS 失败，降级到系统 DNS
        }

        // 3. 降级到系统 DNS
        return Dns.SYSTEM.lookup(hostname)
    }

    private fun parseIps(json: String?): List<InetAddress> {
        // 解析 HTTPDNS 返回的 IP 列表
        return json?.let {
            JSONObject(it).getJSONArray("ips").let { array ->
                (0 until array.length()).map { i ->
                    InetAddress.getByName(array.getString(i))
                }
            }
        } ?: emptyList()
    }
}
```

**3. 数据压缩优化：**

```kotlin
// 请求体 Gzip 压缩
class GzipRequestInterceptor : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val originalRequest = chain.request()
        val body = originalRequest.body ?: return chain.proceed(originalRequest)

        val compressedRequest = originalRequest.newBuilder()
            .header("Content-Encoding", "gzip")
            .method(originalRequest.method, gzip(body))
            .build()

        return chain.proceed(compressedRequest)
    }

    private fun gzip(body: RequestBody): RequestBody {
        return object : RequestBody() {
            override fun contentType() = body.contentType()
            override fun writeTo(sink: BufferedSink) {
                val gzipSink = GzipSink(sink).buffer()
                body.writeTo(gzipSink)
                gzipSink.close()
            }
        }
    }
}
```

**4. 弱网优化策略：**

```kotlin
class WeakNetworkStrategy {
    fun getOptimalConfig(networkQuality: NetworkQuality): RequestConfig {
        return when (networkQuality) {
            NetworkQuality.EXCELLENT -> RequestConfig(
                timeout = 10_000,
                retryCount = 1,
                imageQuality = ImageQuality.HIGH
            )
            NetworkQuality.GOOD -> RequestConfig(
                timeout = 15_000,
                retryCount = 2,
                imageQuality = ImageQuality.MEDIUM
            )
            NetworkQuality.POOR -> RequestConfig(
                timeout = 30_000,
                retryCount = 3,
                imageQuality = ImageQuality.LOW,
                enablePreload = false // 关闭预加载
            )
            NetworkQuality.OFFLINE -> RequestConfig(
                useCache = true,
                showOfflineTip = true
            )
        }
    }

    // 网络质量探测
    fun detectNetworkQuality(): NetworkQuality {
        // 基于最近 N 次请求的 RTT 和成功率判断
        val avgRtt = recentRequests.map { it.rtt }.average()
        val successRate = recentRequests.count { it.success } / recentRequests.size.toFloat()

        return when {
            avgRtt < 200 && successRate > 0.95 -> NetworkQuality.EXCELLENT
            avgRtt < 500 && successRate > 0.8 -> NetworkQuality.GOOD
            successRate > 0.5 -> NetworkQuality.POOR
            else -> NetworkQuality.OFFLINE
        }
    }
}
```
