# 系统服务依赖解决、发布机制与线程模型

## 问题

> 系统服务之间互相依赖怎么解决？系统服务怎么发布让 App 可见？系统服务跑在什么线程上？

## 核心答案

**依赖解决**：四层机制配合——分阶段启动消解顺序依赖、`systemReady()` 延迟激活解决同阶段循环依赖、`LocalServices` 处理同进程轻量依赖、观察者/回调解耦双向依赖。

**发布机制**：通过 `ServiceManager.addService()` 注册 Binder 对象，App 经 `getSystemService()` → `ServiceManager.getService()` 获取 Binder 代理。

**线程模型**：Binder 线程池接收 IPC 请求，通常 post 到主线程 Handler 串行处理业务逻辑，各服务另有自己的 Handler 工作线程。

## 深入解析

### 原理层

#### 一、系统服务之间的依赖解决

**机制一：分阶段启动（启动顺序消解依赖）**

```text
startBootstrapServices()   → 无外部依赖的基础服务（AMS、PMS、PowerManager）
startCoreServices()        → 依赖引导服务的服务（BatteryService 等）
startOtherServices()       → 依赖前两阶段的服务（WMS、InputManager 等）
```

通过启动顺序保证被依赖的服务先创建。

**机制二：延迟注册 + systemReady()（解决同阶段循环依赖）**

AMS 和 WMS 存在循环依赖：
- AMS 需要 WMS 来管理窗口和 Surface
- WMS 需要 AMS 来管理 Activity 栈和 Task

解法：**先实例化并互相注入引用，等所有服务都就绪后再激活**

```text
// startOtherServices() 中：

// Step 1: 创建（构造函数只做最小初始化，不启动复杂逻辑）
WindowManagerService wm = WindowManagerService.main(...);

// Step 2: 互相注入引用（此时不调用对方的复杂方法）
ams.setWindowManager(wm);
wm.setAms(ams);

// Step 3: 所有服务就绪后，统一调用 systemReady()
ams.systemReady(() -> {
    wm.systemReady();
    // 按依赖顺序依次激活
}, ...);
```

`systemReady()` 的语义：
- 标记服务进入正常运行状态
- 恢复之前保存的状态（pending alarms、registered receivers）
- 启动依赖此服务的其他组件
- 发送 `ACTION_BOOT_COMPLETED` 广播

**机制三：LocalServices —— 同进程内轻量依赖**

```text
// SystemServer 内部，不走 Binder，直接 Java 引用
LocalServices.addService(ActivityManagerInternal.class, amsInternal);

// 其他服务获取
ActivityManagerInternal ami = LocalServices.getService(ActivityManagerInternal.class);
ami.someMethod();  // 直接方法调用，无 Binder 开销
```

- 只在 SystemServer 进程内可见
- 用于服务内部紧密协作
- 绕过 ServiceManager 和 Binder，性能好

**机制四：观察者 / 回调通知（解耦双向依赖）**

```text
// PMS 包变化时通知 AMS 更新组件信息
mPackageManager.registerPackageCallback(new PackageCallback() {
    @Override
    public void onPackageChanged(String pkg) {
        // AMS 更新已安装组件信息
    }
});
```

服务不需要直接持有对方引用，通过注册回调实现松耦合。

**四种机制的适用场景：**

| 机制 | 适用场景 | 示例 |
|------|----------|------|
| 分阶段启动 | 启动顺序依赖 | PMS 在 AMS 之前启动 |
| systemReady() | 同阶段循环依赖 | AMS ↔ WMS 互相需要 |
| LocalServices | 同进程内频繁调用 | AMS 内部调用 PowerManagerInternal |
| 观察者/回调 | 异步状态通知 | PMS 包变化通知 AMS |

#### 二、系统服务的发布机制

**两步发布：注册 + 获取**

```text
SystemServer 内部                      App 进程
────────────────                      ──────────

// 注册                               // 获取
SM.addService("activity", ams);       context.getSystemService("activity")
SM.addService("window", wm);            → SM.getService("activity")
SM.addService("package", pms);            → 返回 AMS 的 Binder 代理
```

**App 调用 getSystemService() 的完整链路：**

```text
context.getSystemService(Context.ACTIVITY_SERVICE)
  → SystemServiceRegistry.getSystemService(Context.ACTIVITY_SERVICE)
    → ServiceManager.getService("activity")
      → 通过 ServiceManager 的固定 Binder 引用查询
      → 返回 AMS 的 Binder 代理对象（IActivityManager.Stub.Proxy）
        → App 拿到代理，后续调用走 Binder IPC
```

**SystemServiceRegistry 的缓存机制：**

```text
// SystemServiceRegistry 静态注册表（编译时确定）
// App 进程启动时就注册好了所有 Manager 的创建工厂
static {
    registerService(Context.ACTIVITY_SERVICE, ActivityManager.class,
        new CachedServiceFetcher<ActivityManager>() {
            @Override
            public ActivityManager createService(ContextImpl ctx) {
                return new ActivityManager(ctx, ctx.mMainThread);
            }
        });
    // ... 其他服务
}

// getSystemService() 调用时：
// → 从缓存中取 Manager 对象（每个 App 进程只创建一次）
// → Manager 内部持有 Binder 代理，调用最终走 IPC
```

**ServiceManager 的特殊性：**

```text
ServiceManager（native 进程）
  ├── 由 init 进程启动（不是 SystemServer 内的服务）
  ├── 是 Binder 注册表：维护 "服务名 → Binder引用" 映射
  ├── 自己的 Binder 引用是固定的（handle = 0）
  ├── 所有进程都通过这个固定引用找到 ServiceManager
  └── 作用类似于 DNS：输入服务名，输出 Binder 引用
```

#### 三、系统服务的线程模型

**SystemServer 进程的线程结构：**

```text
SystemServer 进程
  ├── 主线程（main）              → Looper.loop()，运行 Handler 消息循环
  │                                 → 处理有序的业务逻辑
  │                                 → systemReady() 调用
  │
  ├── Binder 线程池（binder:1~N）  → 接收 App 进程的 Binder IPC 调用
  │                                 → 调用服务的 Stub 方法
  │                                 → 通常 post 到主线程 Handler
  │
  ├── android.display 线程         → Display 事件、窗口动画
  ├── android.ui 线程              → UI 相关异步任务
  ├── PowerManagerService.Worker   → 电源管理后台任务
  ├── PackageManager 线程          → 包扫描、dex 优化
  └── ... 各服务自己的 Handler 线程
```

**典型调用流程：**

```text
App 进程                       SystemServer 进程
─────────                      ─────────────────
startActivity()
  → IActivityManager.Stub.Proxy
    → Binder IPC ────────────→  Binder 线程
                                  → AMS.startActivity() [Binder 线程]
                                    → mH.post(() -> {
                                        // 在主线程处理
                                        // 操作 ActivityRecord
                                        // 通知 WMS 等
                                      })
```

**为什么 Binder 请求不在 Binder 线程直接处理？**

- 多个 App 同时发起请求 → 并发问题
- Activity 栈、窗口列表等共享状态需要同步访问
- 在主线程串行处理避免锁竞争
- Binder 线程负责"接电话"，主线程负责"处理业务"

**各服务的 Handler 线程示例：**

```text
AMS：
  ├── main handler        → Activity 生命周期调度
  ├── mBgHandler          → 权限检查、进程统计等
  └── mUiHandler          → 对话框、Toast 等 UI

WMS：
  ├── main handler        → 窗口布局、焦点管理
  ├── android.display     → Display 事件
  └── animation handler   → 窗口动画
```

### 实战层

**面试常见追问：**

- AMS 和 WMS 的循环依赖是怎么解决的？
- `LocalServices` 和 `ServiceManager.getService()` 有什么区别？
- App 的 `getSystemService()` 为什么返回的是 Manager 而不是 Service 本身？
- Binder 线程和主线程之间是什么关系？为什么不直接在 Binder 线程处理？
- SystemServer 中有几种线程？分别做什么？
- 如果一个系统服务处理请求太慢，会影响其他服务吗？
- ServiceManager 本身是怎么启动的？

**Manager vs Service 的关系：**

```text
ActivityManagerService     → SystemServer 内部的服务实现（Binder Stub）
ActivityManager            → App 侧的 Manager 封装（持有 Binder 代理）

App 调用 am.getRunningTasks()
  → ActivityManager 内部调用 IActivityManager.Stub.Proxy.getRunningTasks()
    → Binder IPC → SystemServer 中的 AMS.getRunningTasks()
```

Manager 是 App 侧的封装层，Service 是 SystemServer 中的实现。App 不直接持有 Service 引用，只持有 Manager 和 Binder 代理。

**排查和观察：**

```bash
# 查看 SystemServer 的线程
adb shell ps -T -p $(adb shell pidof system_server)

# 查看 Binder 线程池大小
adb shell cat /proc/$(adb shell pidof system_server)/status | grep Threads

# 查看所有系统服务
adb shell service list

# 查看某个服务的 Binder 调用统计
adb shell dumpsys binder_calls_stats | grep ActivityManager
```

**源码入口：**

- `SystemServer.java`：三阶段启动 + systemReady 调用顺序
- `ServiceManager.java`：`addService()` / `getService()` 注册和查询
- `SystemServiceRegistry.java`：App 侧 Manager 缓存
- `LocalServices.java`：同进程轻量服务注册
- `ActivityManagerService.java`：`systemReady()` 实现

**容易答错的点：**

- 系统服务不是"并行独立运行"的，它们有复杂的依赖关系，通过多种机制解决
- `systemReady()` 不是构造函数的一部分，它是延迟激活，保证依赖都就绪后才开始工作
- App 拿到的不是 Service 本身，是 Manager 封装 + Binder 代理
- Binder 线程和主线程不是同一个线程，Binder 请求通常 post 到主线程串行处理
- ServiceManager 不是 SystemServer 内的服务，它是独立的 native 进程
