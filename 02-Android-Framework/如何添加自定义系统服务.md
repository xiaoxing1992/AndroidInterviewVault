# 如何添加自定义系统服务

## 问题

> 如何在 Android 中添加一个自定义系统服务？需要改哪些文件？完整流程是什么？

## 核心答案

添加系统服务需要六步：定义 AIDL 接口（跨进程通信契约）→ 实现 Service 类（继承 Stub，在 SystemServer 中运行）→ 编写 Manager 类（App 侧封装）→ 注册到 SystemServiceRegistry（让 `getSystemService()` 能查到）→ 在 Context 中添加服务名常量 → 在 SystemServer 中启动（addService + systemReady）。此外还需要配置 SELinux 策略和编译脚本。

## 深入解析

### 原理层

**各层的对应关系：**

```text
App 进程                              SystemServer 进程
──────                                ─────────────────
DeviceMonitorManager                  DeviceMonitorManager（实现类）
  → IDeviceMonitorManager.Stub.Proxy    → IDeviceMonitorManager.Stub
    → Binder IPC ──────────────────→      → 处理请求

context.getSystemService("device_monitor")
  → SystemServiceRegistry
    → ServiceManager.getService("device_monitor")
      → 返回 Binder 代理 → 包装成 Manager
```

**完整六步流程：**

**Step 1: 定义 AIDL 接口**

```java
// frameworks/base/core/java/android/app/IDeviceMonitorManager.aidl
package android.app;

/** @hide */
interface IDeviceMonitorManager {
    String getDeviceInfo();
    void registerCallback(IDeviceMonitorCallback callback);
}
```

AIDL 编译自动生成：
- `IDeviceMonitorManager.Stub` — 服务端基类（Binder 实现）
- `IDeviceMonitorManager.Stub.Proxy` — 客户端代理

**Step 2: 实现 Service 类**

```java
// frameworks/base/services/core/java/com/android/server/DeviceMonitorService.java
package com.android.server;

public class DeviceMonitorService extends IDeviceMonitorManager.Stub {

    private final Context mContext;
    private final Handler mHandler;
    private volatile boolean mSystemReady = false;

    public DeviceMonitorService(Context context) {
        mContext = context;
        mHandler = new Handler(Looper.getMainLooper());
    }

    @Override  // Binder 线程调用
    public String getDeviceInfo() {
        return "device info";
    }

    @Override
    public void registerCallback(IDeviceMonitorCallback cb) {
        // 注册回调
    }

    // 延迟激活
    public void systemReady() {
        mSystemReady = true;
        startMonitoring();
    }
}
```

**Step 3: 编写 Manager 类（App 侧封装）**

```java
// frameworks/base/core/java/android/app/DeviceMonitorManager.java
package android.app;

/** @hide */
public class DeviceMonitorManager {

    private final IDeviceMonitorManager mService;

    /** @hide */
    public DeviceMonitorManager(IDeviceMonitorManager service) {
        mService = service;
    }

    public String getDeviceInfo() {
        try {
            return mService.getDeviceInfo();
        } catch (RemoteException e) {
            throw e.rethrowFromSystemServer();
        }
    }
}
```

**Step 4: 注册到 SystemServiceRegistry**

```java
// frameworks/base/core/java/android/app/SystemServiceRegistry.java
// 在 static {} 块中添加：

registerService(Context.DEVICE_MONITOR_SERVICE, DeviceMonitorManager.class,
    new CachedServiceFetcher<DeviceMonitorManager>() {
        @Override
        public DeviceMonitorManager createService(ContextImpl ctx) {
            IBinder binder = ServiceManager.getService(Context.DEVICE_MONITOR_SERVICE);
            IDeviceMonitorManager service =
                IDeviceMonitorManager.Stub.asInterface(binder);
            return new DeviceMonitorManager(service);
        }
    });
```

`CachedServiceFetcher` 保证每个 App 进程只创建一次 Manager 实例（缓存在 ContextImpl 中）。

**Step 5: 在 Context 中添加常量**

```java
// frameworks/base/core/java/android/content/Context.java
/** @hide */
public static final String DEVICE_MONITOR_SERVICE = "device_monitor";
```

**Step 6: 在 SystemServer 中启动**

```java
// frameworks/base/services/java/com/android/server/SystemServer.java

private DeviceMonitorService mDeviceMonitorService;

private void startOtherServices(SystemServicesAtm atm) {
    // 创建并注册到 ServiceManager
    mDeviceMonitorService = new DeviceMonitorService(mSystemContext);
    ServiceManager.addService(
        Context.DEVICE_MONITOR_SERVICE, mDeviceMonitorService);

    // 延迟激活
    mActivityManagerService.systemReady(() -> {
        // ...
        mDeviceMonitorService.systemReady();
        // ...
    }, ...);
}
```

**还需要配置的额外项：**

**SELinux 策略：**

```text
# service_contexts — 给服务添加安全标签
device_monitor                  u:object_r:device_monitor_service:s0

# device_monitor_service.te — 定义服务的 SELinux 类型
type device_monitor_service, system_api_service, system_server_service, service_manager_type;

# App 访问权限（按需配置）
allow untrusted_app device_monitor_service:service_manager find;
```

**编译配置：**

```text
# Android.bp（frameworks/base 中）
// AIDL 需要声明为 source
java_library {
    name: "framework",
    srcs: [
        "core/java/android/app/IDeviceMonitorManager.aidl",
        // ...
    ],
}
```

**App 调用方式：**

```java
// App 代码
DeviceMonitorManager dm = (DeviceMonitorManager)
    context.getSystemService(Context.DEVICE_MONITOR_SERVICE);
String info = dm.getDeviceInfo();
```

### 实战层

**面试常见追问：**

- AIDL 接口为什么是必须的？直接用 Binder 行不行？
- Manager 和 Service 的关系是什么？为什么不直接暴露 Service？
- 为什么要注册到 SystemServiceRegistry？
- SELinux 策略如果不配置会怎样？
- 自定义系统服务可以不放在 SystemServer 中吗？
- 如何给系统服务添加权限检查？

**Manager vs Service 的区别：**

| | Manager（App 侧） | Service（SystemServer 侧） |
|---|---|---|
| 位置 | App 进程 | SystemServer 进程 |
| 作用 | 封装 Binder 调用，提供简洁 API | 实现业务逻辑 |
| 继承 | 普通 Java 类 | IxxxManager.Stub |
| 线程 | App 主线程 | Binder 线程 → Handler 主线程 |

**不注册 SystemServiceRegistry 的后果：**

App 调用 `context.getSystemService("device_monitor")` 会返回 `null`。App 需要用更底层的方式获取：

```java
// 不推荐的写法（绕过 SystemServiceRegistry）
IBinder binder = ServiceManager.getService("device_monitor");
IDeviceMonitorManager service = IDeviceMonitorManager.Stub.asInterface(binder);
```

**权限检查模式：**

```java
@Override
public String getDeviceInfo() {
    // 权限检查
    mContext.enforceCallingOrSelfPermission(
        "android.permission.DEVICE_MONITOR",
        "Need DEVICE_MONITOR permission");

    // 或者基于 UID 检查
    int callingUid = Binder.getCallingUid();
    if (callingUid != Process.SYSTEM_UID) {
        throw new SecurityException("Only system can call");
    }

    return "info";
}
```

**排查和观察：**

```bash
# 查看新服务是否注册成功
adb shell service list | grep device_monitor

# 查看服务的 Binder 对象
adb shell service check device_monitor

# 查看 SELinux 拒绝日志（如果策略没配）
adb logcat | grep "avc: denied"
```

**容易答错的点：**

- AIDL 不是可选的，跨进程通信必须用 AIDL 定义接口（或手写 Binder）
- Manager 不是 Service 的子类，它们在不同进程，Manager 只持有 Binder 代理
- 不注册 SystemServiceRegistry 也能工作，但 App 就不能用 `getSystemService()` 了
- SELinux 策略不配置的话，服务注册到 ServiceManager 会被 SELinux 拦截
- 系统服务也可以不放在 SystemServer 中，但通常应该放（共享 system_server 的权限和生命周期）

## 记忆锚点

添加系统服务六步走：**AIDL 接口 → Service 实现 → Manager 封装 → SystemServiceRegistry 注册 → Context 常量 → SystemServer 启动**。外加 SELinux 策略和编译配置。本质是把一个 Binder 对象从 SystemServer 暴露到 App 进程。
