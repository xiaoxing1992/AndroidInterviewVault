# Android 系统服务启动流程

## 问题

> Android 的系统服务（AMS、PMS、WMS 等）是怎么启动的？启动顺序是什么？为什么要分三个阶段？

## 核心答案

系统服务在 `SystemServer.run()` 中分三阶段启动：`startBootstrapServices()` 启动引导服务（AMS、PMS 等）、`startCoreServices()` 启动核心服务（BatteryService 等）、`startOtherServices()` 启动其他服务（WMS、InputManager 等）。每个服务的启动流程是：实例化 → `ServiceManager.addService()` 注册到 Binder 注册表 → `systemReady()` 就绪。分阶段是因为服务间有严格依赖关系，必须保证被依赖的服务先启动。启动完成后，其他进程通过 `ServiceManager.getService()` 获取 Binder 引用来调用系统服务。

## 深入解析

### 原理层

**SystemServer.run() 总览：**

```java
// SystemServer.java（简化）
private void run() {
    // 准备阶段
    Looper.prepareMainLooper();
    System.loadLibrary("android_servers");

    // ★ 三阶段启动服务
    startBootstrapServices(t);   // 引导服务
    startCoreServices(t);        // 核心服务
    startOtherServices(t);       // 其他服务

    // 进入消息循环，永不返回
    Looper.loop();
}
```

**阶段一：startBootstrapServices() —— 引导服务**

这些是其他服务依赖的基础服务，必须最先启动：

```text
startBootstrapServices():
  → Installer                    // 包安装辅助（apk 安装、dex 优化）
  → DeviceIdentsPolicyService    // 设备标识策略
  → UriGrantsManagerService      // URI 权限管理
  → ActivityManagerService       // ★ AMS — 进程管理、四大组件调度
  → PowerManagerService          // ★ 电源管理
  → RecoverySystemService        // 恢复模式
  → PackageManagerService        // ★ PMS — 包管理、权限、组件注册
  → UserManagerService           // 多用户管理
  → OverlayManagerService        // 资源覆盖
  → ...
```

关键依赖：
- PMS 必须在 AMS 之后、其他服务之前（AMS 启动时要查 PMS 解析 intent 和组件）
- PowerManagerService 必须在 WMS 之前（WMS 依赖电源状态）

**阶段二：startCoreServices() —— 核心服务**

不是基础依赖，但属于系统核心功能：

```text
startCoreServices():
  → BatteryService               // 电池状态和充电信息
  → UsageStatsService            // 应用使用统计
  → WebViewUpdateService         // WebView 引擎更新
  → CachedDeviceStateService     // 设备缓存状态
  → BinderCallsStatsService      // Binder 调用统计
  → ...
```

**阶段三：startOtherServices() —— 其他服务**

最多的服务集中在这个阶段，因为它们依赖前两阶段的服务：

```text
startOtherServices():
  → WindowManagerService         // ★ WMS — 窗口管理、Surface 分配
  → InputManagerService          // ★ 输入事件分发
  → ConnectivityService          // 网络连接管理
  → AlarmManagerService          // 闹钟和定时器
  → JobSchedulerService          // 后台任务调度
  → NotificationManagerService   // 通知管理
  → LocationManagerService       // 定位服务
  → AudioService                 // 音频管理
  → ...
```

**服务间的依赖关系图：**

```text
PMS ──────────┐
              ├──→ AMS ──→ WMS ──→ InputManager
PowerManager ─┘         │
                        ├──→ ActivityStarter
                        └──→ DisplayManager

Installer ──→ PMS（PMS 需要 Installer 执行 dex 优化和包安装）

AMS ──→ 大部分其他服务（很多服务需要注册 AMS 回调）
```

**每个服务的启动三步曲：**

```text
// 以 WindowManagerService 为例

// Step 1: 实例化
WindowManagerService wm = WindowManagerService.main(context, inputManager, ...);
    → 初始化 DisplayContent
    → 创建 InputMonitor
    → 准备窗口策略

// Step 2: 注册到 ServiceManager
ServiceManager.addService(Context.WINDOW_SERVICE, wm, /* allowIsolated= */ false,
        DUMP_FLAG_PRIORITY_CRITICAL | DUMP_FLAG_PROTO);
    → 向 ServiceManager 注册 Binder 对象
    → 其他进程现在可以通过 SM.getService("window") 拿到 WMS 的代理

// Step 3: 就绪回调
wm.systemReady();
    → 标记服务就绪
    → 开始处理来自其他进程的请求
    → 可能触发依赖此服务的其他组件启动
```

**ServiceManager 在服务启动中的角色：**

```text
                    ServiceManager（Binder 注册表）
                    ┌──────────────────────────────────┐
                    │ "activity" → AMS 的 Binder 对象    │
SystemServer        │ "package"  → PMS 的 Binder 对象    │    App 进程
  SM.addService() → │ "window"   → WMS 的 Binder 对象    │ ← SM.getService()
                    │ "alarm"    → AlarmMS 的 Binder 对象 │
                    │ ...                                │
                    └──────────────────────────────────┘
```

- `ServiceManager.addService()` 将服务的 Binder 对象注册到 ServiceManager
- 其他进程通过 `ServiceManager.getService("name")` 获取 Binder 代理
- 之后的跨进程调用走 Binder IPC，直接到达 SystemServer 中的目标服务

**systemReady() 的作用：**

很多服务启动后不会立即处理请求，而是等到 `systemReady()` 被调用后才真正就绪：

```text
AMS.systemReady() 的触发时机：
  → 所有引导服务和核心服务启动完成
  → 系统UI 启动
  → 所有服务的 systemReady() 被依次调用
  → 发送 ACTION_BOOT_COMPLETED 广播
  → Home Activity 启动
```

`systemReady()` 通常做：
- 标记服务进入正常运行状态
- 启动依赖此服务的其他组件
- 恢复之前保存的状态（如 pending alarms、registered receivers）

**服务数量和注册表：**

AOSP 中有上百个系统服务，可以通过以下方式查看：

```bash
# 列出所有注册的系统服务
adb shell service list

# 查看某个服务的 Binder 对象
adb shell service check activity
adb shell service check window
```

### 实战层

**面试常见追问：**

- `startBootstrapServices()` 里 AMS 和 PMS 的启动顺序是什么？为什么？
- WMS 为什么放在 `startOtherServices()` 而不是 `startBootstrapServices()`？
- `ServiceManager.addService()` 和 `getSystemService()` 的关系是什么？
- `systemReady()` 有什么作用？所有服务都有这个方法吗？
- 系统服务是在哪个线程启动的？是并行还是串行？
- 系统服务崩了会怎样？和 SystemServer 崩了有什么区别？
- 如何添加一个自定义系统服务？

**系统服务的启动方式：**

所有系统服务在 SystemServer 的**主线程**中**串行**启动，不是并行。原因是：
- 服务间有严格的依赖关系
- 并行启动难以保证初始化顺序
- 部分服务的构造函数会调用其他已启动的服务

**添加自定义系统服务的步骤：**

```text
1. 定义 AIDL 接口（IXxxManager.aidl）
2. 实现服务类（XxxManagerService extends IXxxManager.Stub）
3. 在 SystemServer.startOtherServices() 中添加启动代码
4. 在 Context 中添加服务名常量
5. 在 ServiceManager 中注册
6. 编写 Manager 类供 App 调用（如 XxxManager.java）
```

推荐阅读顺序：
1. `SystemServer.java` `run()` → `startBootstrapServices()` —— 看三阶段结构和启动顺序
2. `ActivityManagerService` 构造函数 —— 看 AMS 初始化了什么
3. `ServiceManager.addService()` —— 看 Binder 注册机制
4. `WindowManagerService.main()` —— 看一个具体服务的启动过程

**容易答错的点：**

- 系统服务不是并行启动的，是串行的，因为在 SystemServer 主线程中依次调用
- `systemReady()` 不是构造函数的一部分，它是后续单独调用的，表示服务可以开始处理请求
- ServiceManager 不是系统服务的容器，它是 Binder 注册表（名字 → Binder 引用的映射）
- 系统服务崩了不等于 SystemServer 崩了，单个服务异常通常被 try-catch 捕获
- PMS 不只管包安装，还管权限、intent 解析、组件注册

## 记忆锚点

系统服务在 `SystemServer.run()` 中**串行三阶段**启动：引导（AMS/PMS）→ 核心（Battery）→ 其他（WMS）。每个服务三步走：实例化 → `SM.addService()` 注册 → `systemReady()` 就绪。ServiceManager 是 Binder 注册表，不是服务容器。
