# 系统服务获取机制

## 问题

> App 是怎么获取系统服务的？getSystemService() 内部经历了什么？有哪些获取方式？

## 核心答案

App 获取系统服务有三条路径：`getSystemService()`（最常用，经过 SystemServiceRegistry 缓存 + ServiceManager 查询）、`ServiceManager.getService()`（底层直接获取 Binder 代理）、`LocalServices.getService()`（SystemServer 内部同进程调用，不走 Binder）。`getSystemService()` 的核心设计是**静态注册表 + 进程级缓存**——每个 App 进程只和 ServiceManager 交互一次，之后全部缓存命中。

## 深入解析

### 原理层

**三条获取路径对比：**

| 路径 | 适用场景 | 走 Binder？ | 有缓存？ |
|------|----------|-------------|---------|
| `getSystemService()` | App 获取服务 | 是 | 有（CachedServiceFetcher） |
| `ServiceManager.getService()` | 系统内部直接获取代理 | 是 | 无（每次查 SM） |
| `LocalServices.getService()` | SystemServer 内部互调 | 否（直接引用） | 无 |

#### 路径一：getSystemService()（最常用）

**完整调用链：**

```text
context.getSystemService(Context.ACTIVITY_SERVICE)
  → ContextImpl.getSystemService(name)
    → SystemServiceRegistry.getSystemService(name, ContextImpl)
      → 从静态注册表 SYSTEM_SERVICE_FETCHERS 取 ServiceFetcher
        → CachedServiceFetcher.getService()
          → 缓存命中？直接返回 Manager
          → 未命中？
            → ServiceManager.getService("activity")   // Binder IPC 查 SM
            → 返回 AMS 的 Binder 代理（IBinder）
            → new ActivityManager(binderProxy)          // 包装成 Manager
            → 存入缓存，返回
```

**SystemServiceRegistry —— 静态注册表：**

```java
// 每个 App 进程一份，类加载时执行 static {}
public final class SystemServiceRegistry {

    // 注册表：服务名 → ServiceFetcher
    private static final HashMap<String, ServiceFetcher<?>>
        SYSTEM_SERVICE_FETCHERS = new HashMap<>();

    // 注册（编译时确定，AOSP 中有上百个）
    private static <T> void registerService(String name,
            Class<T> clazz, ServiceFetcher<T> fetcher) {
        SYSTEM_SERVICE_FETCHERS.put(name, fetcher);
    }

    // 获取
    public static Object getSystemService(String name) {
        ServiceFetcher<?> f = SYSTEM_SERVICE_FETCHERS.get(name);
        return f != null ? f.getService(contextImpl) : null;
    }

    // 静态初始化块
    static {
        registerService(Context.ACTIVITY_SERVICE, ActivityManager.class,
            new CachedServiceFetcher<ActivityManager>() {
                @Override
                public ActivityManager createService(ContextImpl ctx) {
                    IBinder b = ServiceManager.getService(Context.ACTIVITY_SERVICE);
                    IActivityManager am = IActivityManager.Stub.asInterface(b);
                    return new ActivityManager(am);
                }
            });
        registerService(Context.WINDOW_SERVICE, WindowManager.class,
            new CachedServiceFetcher<WindowManager>() { ... });
        registerService(Context.PACKAGE_SERVICE, PackageManager.class,
            new CachedServiceFetcher<PackageManager>() { ... });
        // ... 上百个服务
    }
}
```

**缓存机制：**

```text
CachedServiceFetcher 的缓存策略：

  App 进程启动
    → SystemServiceRegistry 类加载
    → 注册表建立（name → ServiceFetcher 映射）

  第一次 getSystemService("activity"):
    → CachedServiceFetcher.getService()
      → 缓存未命中
      → createService() → ServiceManager.getService() → Binder IPC
      → 创建 Manager，存入 ContextImpl.mServiceCache[id]
      → 返回 Manager

  第二次 getSystemService("activity"):
    → CachedServiceFetcher.getService()
      → 缓存命中，直接返回 mServiceCache[id]
      → 无 Binder IPC，极快
```

CachedServiceFetcher vs StaticServiceFetcher：

| 类型 | 缓存方式 | 适用场景 |
|------|----------|----------|
| CachedServiceFetcher | 存在 ContextImpl 的 mServiceCache 数组中，每个 Context 一份 | 大部分服务 |
| StaticServiceFetcher | static 变量缓存，进程内全局单例 | 少数全局唯一的服务 |

#### 路径二：ServiceManager.getService()（底层直接获取）

```java
// 不经过 SystemServiceRegistry，直接获取 Binder 代理
IBinder binder = ServiceManager.getService("activity");
IActivityManager am = IActivityManager.Stub.asInterface(binder);
```

- 返回 IBinder / IInterface，不是 Manager
- 没有缓存，每次调用都会查 ServiceManager
- 系统内部代码和需要直接操作 Binder 的场景使用

**ServiceManager.getService() 内部过程：**

```text
ServiceManager.getService("activity")
  → sServiceManager.getService("activity")   // 懒加载的 SM 代理
    → Binder IPC 到 ServiceManager 进程
      → SM 内部：HashMap<String, IBinder>.get("activity")
      → 返回 AMS 的 Binder 引用
    ← 返回 IBinder
```

**ServiceManager 自身怎么被发现？**

```text
ServiceManager 的 Binder handle = 0（硬编码）
  → 所有进程都知道 handle 0 就是 ServiceManager
  → 不需要"查找 ServiceManager"这个过程
  → Binder 驱动的特殊设计
```

#### 路径三：LocalServices.getService()（同进程内）

```java
// 只在 SystemServer 进程内可用
ActivityManagerInternal ami =
    LocalServices.getService(ActivityManagerInternal.class);
```

- 不经过 ServiceManager、不经过 Binder
- 直接 Java 引用调用
- 用于 SystemServer 内部服务之间的紧密协作

### 完整链路图

```text
App 进程                                    SystemServer 进程
────────                                    ─────────────────

context.getSystemService("activity")
  │
  ▼
SystemServiceRegistry
  │  缓存？──→ 命中 → 返回 Manager（无 IPC）
  │  未命中
  ▼
ServiceManager.getService("activity")
  │
  ▼
Binder IPC ──────────────────────→  ServiceManager 进程（handle=0）
                                      │  HashMap.get("activity")
                                      │  返回 AMS 的 Binder 引用
  ←─────────────────────────────────  │
  │
  ▼
new ActivityManager(binderProxy)    AMS（Binder Stub）
  │                                   │
  │  缓存到 mServiceCache             │
  ▼                                   ▼
App 拿到 Manager               SystemServer 中运行的 AMS
  │
  ▼
am.startActivity(intent)
  → IActivityManager.Stub.Proxy.startActivity()
    → Binder IPC ──────────────→  AMS.startActivity()
                                    → mH.post(() -> { 主线程处理 })
```

### 实战层

**面试常见追问：**

- `getSystemService()` 为什么快？缓存在哪？
- `ServiceManager.getService()` 和 `getSystemService()` 的区别？
- SystemServiceRegistry 是每个 App 进程一份还是全局一份？
- ServiceManager 的 Binder handle 为什么是 0？
- `getSystemService()` 返回的是 Service 还是 Manager？
- 如果 ServiceManager 挂了会怎样？

**getSystemService() 返回的是什么？**

```text
返回的是 Manager（App 侧封装类），不是 Service（SystemServer 中的实现）

ActivityManager am = getSystemService(ActivityManager.class);
  → am 是 ActivityManager 对象（App 进程中）
  → am 内部持有 IActivityManager.Stub.Proxy（Binder 代理）
  → am.startActivity() 实际调用的是 proxy.startActivity()
    → Binder IPC → SystemServer 中的 AMS.startActivity()
```

**如果 ServiceManager 挂了：**

```text
ServiceManager 是系统关键进程，如果崩溃：
  → 所有 getService() 调用失败
  → 无法发现新的系统服务
  → 已经拿到的 Binder 代理不受影响（Binder 引用独立于 SM）
  → init 进程会重启 ServiceManager
```

**排查和观察：**

```bash
# 列出所有注册的系统服务
adb shell service list

# 查看某个服务是否可用
adb shell service check activity

# 查看 SystemServiceRegistry 中注册了多少服务
# 通过代码：SystemServiceRegistry 的 static 块

# 查看 App 进程的 Binder 连接
adb shell ls -la /proc/$(adb shell pidof com.example.app)/fd | grep binder
```

**容易答错的点：**

- `getSystemService()` 返回的是 Manager，不是 Service 本身
- SystemServiceRegistry 是每个 App 进程一份（静态类），不是全局共享
- 第一次 `getSystemService()` 有 Binder IPC 开销，之后缓存命中没有
- ServiceManager 的 handle=0 是硬编码，不是运行时分配的
- `LocalServices` 只在 SystemServer 内部可用，App 进程不能用
