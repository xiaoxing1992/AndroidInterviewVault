# 热修复原理 - 类加载与底层替换

## 问题

> Android 热修复的主流方案有哪些？类加载方案和底层替换方案的原理分别是什么？各有什么优缺点？

## 核心答案

热修复两大流派：类加载方案（Tinker/QFix）— 将修复后的 dex 插入 ClassLoader 的 dexElements 数组前面，下次加载类时优先加载修复版本，需要重启生效；底层替换方案（AndFix/Sophix）— 直接修改 ArtMethod 结构体中的方法指针，指向修复后的方法，即时生效无需重启。Sophix 结合两种方案取长补短。

## 深入解析

### 原理层

**方案1: 类加载（Tinker）：**
```
原理: Android ClassLoader 加载类时遍历 dexElements 数组
     将包含修复类的 dex 插入数组最前面
     → 修复类先被加载，原始类不再加载

PathClassLoader
  └── pathList (DexPathList)
        └── dexElements[] (Element[])
              ├── [0] patch.dex  ← 修复的 dex（插入到最前面）
              ├── [1] classes.dex
              ├── [2] classes2.dex
              └── ...

// 实现
val pathList = classLoader.getField("pathList")
val dexElements = pathList.getField("dexElements")
val patchElement = makeDexElement(patchDexFile)
val newElements = arrayOf(patchElement) + dexElements
pathList.setField("dexElements", newElements)
```

**Tinker 的差量方案：**
```
1. 生成差量包: 对比新旧 dex，生成 patch（BsDiff 算法）
2. 下发 patch 文件（几十 KB ~ 几百 KB）
3. 客户端合成: 旧 dex + patch → 新 dex（后台线程）
4. 重启后加载新 dex
```

**方案2: 底层替换（AndFix/Sophix）：**
```
原理: ART 虚拟机中每个方法对应一个 ArtMethod 结构体
     直接替换 ArtMethod 的关键字段（entry_point、dex_code_item_offset）
     → 调用该方法时执行修复后的代码

// ArtMethod 结构体（简化）
struct ArtMethod {
    uint32_t declaring_class_;
    uint32_t access_flags_;
    uint32_t dex_code_item_offset_;  // 方法体在 dex 中的偏移
    void* entry_point_from_quick_compiled_code_;  // 编译后的机器码入口
    // ...
};

// 替换: 将旧方法的 ArtMethod 内容整体替换为新方法的
memcpy(oldMethod, newMethod, sizeof(ArtMethod));
```

**两种方案对比：**

| 维度 | 类加载（Tinker） | 底层替换（AndFix） |
|------|-----------------|-------------------|
| 生效时机 | 重启后 | 即时 |
| 修复范围 | 类级别（可改结构） | 方法级别（不能改类结构） |
| 兼容性 | 高（不依赖虚拟机内部结构） | 低（ArtMethod 结构随版本变化） |
| 补丁大小 | 较大（dex 差量） | 小（方法级别） |
| 性能影响 | 首次加载稍慢 | 无 |
| 资源修复 | 支持 | 不支持 |

**Sophix 混合方案：**
```
小修复（方法级别）→ 底层替换，即时生效
大修复（类结构变化）→ 类加载，重启生效
资源修复 → 替换 AssetManager 中的资源路径
So 修复 → 替换 System.loadLibrary 的加载路径
```

### 实战层

**Tinker 接入：**
```kotlin
// Application 中初始化
@DefaultLifecycleObserver
class TinkerManager {
    fun install(app: Application) {
        TinkerInstaller.install(
            ApplicationLike(app),
            DefaultLoadReporter(app),
            DefaultPatchReporter(app),
            DefaultPatchListener(app)
        )
    }
    
    fun applyPatch(patchPath: String) {
        TinkerInstaller.onReceiveUpgradePatch(context, patchPath)
    }
}
```

**热修复的限制：**
- 不能修复 Application 类（加载时机太早）
- AndroidManifest 变更无法热修复（需要重新安装）
- 新增四大组件无法热修复
- Android 14+ 对 dex 加载有更严格的限制

**选型建议：**
- 紧急 bug 修复（即时生效）→ Sophix 底层替换模式
- 功能修复（可接受重启）→ Tinker
- Google Play 上架 → 不允许热修复（违反政策）

## 记忆锚点

热修复两条路：类加载方案 = 把修复 dex 插到前面（重启生效，兼容性好）；底层替换 = 直接改方法指针（即时生效，兼容性差）。Sophix 两者结合。
