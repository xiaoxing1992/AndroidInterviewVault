# 大数据量分页查询优化

## 问题

> 数据库中有百万级数据，如何高效分页查询？OFFSET 分页有什么问题？Cursor-based 分页是什么？Room 中如何实现高效分页？

## 核心答案

传统 OFFSET 分页在深页时性能急剧下降（需要跳过前 N 行），时间复杂度 O(offset+limit)。Cursor-based（游标）分页使用上一页最后一条记录的排序字段值作为起点，配合索引实现 O(limit) 的稳定性能。Room 中通过 Paging3 + PagingSource 实现，底层使用 LIMIT+WHERE 代替 OFFSET。

## 深入解析

### 原理层

**OFFSET 分页的问题：**
```sql
-- 第 1 页（快）
SELECT * FROM articles ORDER BY created_at DESC LIMIT 20 OFFSET 0;

-- 第 1000 页（慢！）
SELECT * FROM articles ORDER BY created_at DESC LIMIT 20 OFFSET 19980;
-- SQLite 需要扫描前 19980 行然后丢弃，只返回 20 行
-- 越往后越慢，O(offset + limit)
```

**Cursor-based 分页（Keyset Pagination）：**
```sql
-- 第 1 页
SELECT * FROM articles ORDER BY created_at DESC LIMIT 20;
-- 记录最后一条的 created_at = 1700000000

-- 第 2 页（用上一页最后的值作为游标）
SELECT * FROM articles 
WHERE created_at < 1700000000 
ORDER BY created_at DESC LIMIT 20;
-- 直接通过索引定位，O(limit)，与页码无关
```

**性能对比：**
```
数据量: 100万行，索引: created_at

| 页码 | OFFSET 耗时 | Cursor 耗时 |
|------|-------------|-------------|
| 1    | 2ms         | 2ms         |
| 100  | 50ms        | 2ms         |
| 1000 | 500ms       | 2ms         |
| 10000| 5000ms      | 2ms         |

Cursor 分页性能恒定，不受页码影响
```

**处理排序字段重复的情况：**
```sql
-- 如果 created_at 有重复值，需要加上唯一字段（如 id）作为 tiebreaker
SELECT * FROM articles 
WHERE (created_at, id) < (1700000000, 12345)
ORDER BY created_at DESC, id DESC 
LIMIT 20;

-- 或者用 OR 条件（SQLite 兼容写法）
SELECT * FROM articles 
WHERE created_at < 1700000000 
   OR (created_at = 1700000000 AND id < 12345)
ORDER BY created_at DESC, id DESC 
LIMIT 20;
```

**Room + Paging3 实现：**
```kotlin
// Room 自动生成的 PagingSource 使用 LIMIT + OFFSET
// 但 Paging3 的 LoadParams 支持 key-based 分页

@Dao
interface ArticleDao {
    // 方式1: Room 自动生成 PagingSource（内部用 OFFSET，但有 InvalidationTracker 优化）
    @Query("SELECT * FROM articles ORDER BY created_at DESC")
    fun pagingSource(): PagingSource<Int, Article>
    
    // 方式2: 手动 Cursor-based 查询
    @Query("""
        SELECT * FROM articles 
        WHERE created_at < :cursor 
        ORDER BY created_at DESC 
        LIMIT :limit
    """)
    suspend fun getArticlesAfter(cursor: Long, limit: Int): List<Article>
}

// 自定义 PagingSource 使用 Cursor-based
class ArticlePagingSource(private val dao: ArticleDao) : PagingSource<Long, Article>() {
    override suspend fun load(params: LoadParams<Long>): LoadResult<Long, Article> {
        val cursor = params.key ?: Long.MAX_VALUE
        val articles = dao.getArticlesAfter(cursor, params.loadSize)
        return LoadResult.Page(
            data = articles,
            prevKey = null,
            nextKey = articles.lastOrNull()?.createdAt
        )
    }
}
```

### 实战层

**索引设计：**
```kotlin
// 分页查询的索引必须覆盖 ORDER BY 和 WHERE 条件
@Entity(indices = [Index(value = ["created_at", "id"])])
data class Article(
    @PrimaryKey val id: Long,
    val createdAt: Long,
    // ...
)
```

**选型建议：**
- 数据量 < 1 万：OFFSET 分页足够
- 数据量 > 1 万 + 需要深翻页：Cursor-based
- 只需要"加载更多"（无跳页）：Cursor-based 最优
- 需要跳到任意页码：OFFSET 不可避免，但可以缓存 cursor 断点

**其他优化手段：**
- 延迟关联：先查 id 列表（走覆盖索引），再 IN 查询完整数据
- 预计算：定期物化分页结果到缓存表
- 分区：按时间分表，只查询相关分区

## 记忆锚点

OFFSET 分页 = 数页码翻书（越往后越慢，要跳过前面所有页）。Cursor 分页 = 书签定位（直接翻到上次看的地方，恒定速度）。
