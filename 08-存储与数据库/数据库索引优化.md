# 数据库索引优化

## 问题

> SQLite 索引的原理是什么？什么时候应该加索引？复合索引的列顺序有什么讲究？如何分析慢查询？

## 核心答案

SQLite 索引使用 B-Tree 结构，将索引列的值有序存储，查询时通过二分查找快速定位行，避免全表扫描。应该对 WHERE、JOIN、ORDER BY 中频繁使用的列加索引。复合索引遵循最左前缀原则，列顺序应按选择性从高到低排列。通过 `EXPLAIN QUERY PLAN` 分析查询是否使用了索引。

## 深入解析

### 原理层

**B-Tree 索引结构：**
```
索引 B-Tree:
         [30 | 60]              ← 根节点
        /    |    \
   [10|20] [40|50] [70|80]     ← 叶节点（存储 rowid 指针）

查询 WHERE age = 40:
  根节点: 30 < 40 < 60 → 中间子树
  叶节点: 找到 40 → 获取 rowid → 回表查完整行
  
时间复杂度: O(log N) vs 全表扫描 O(N)
```

**Room 中创建索引：**
```kotlin
@Entity(
    tableName = "orders",
    indices = [
        Index(value = ["user_id"]),                    // 单列索引
        Index(value = ["status", "created_at"]),       // 复合索引
        Index(value = ["order_no"], unique = true)     // 唯一索引
    ]
)
data class Order(
    @PrimaryKey val id: Long,
    val userId: Long,
    val status: String,
    val createdAt: Long,
    val orderNo: String
)
```

**复合索引最左前缀原则：**
```sql
-- 索引: (status, created_at)

-- ✅ 使用索引
SELECT * FROM orders WHERE status = 'paid'
SELECT * FROM orders WHERE status = 'paid' AND created_at > 1000
SELECT * FROM orders WHERE status = 'paid' ORDER BY created_at

-- ❌ 不使用索引（跳过了最左列）
SELECT * FROM orders WHERE created_at > 1000
SELECT * FROM orders ORDER BY created_at
```

**索引列顺序选择：**
```
原则: 等值条件列在前，范围条件列在后

-- 查询: WHERE status = ? AND created_at > ? ORDER BY created_at
-- 最优索引: (status, created_at)
-- 原因: status 等值匹配缩小范围，created_at 范围查询+排序利用索引有序性
```

**EXPLAIN QUERY PLAN 分析：**
```sql
EXPLAIN QUERY PLAN SELECT * FROM orders WHERE user_id = 123;

-- 好的结果（使用索引）:
-- SEARCH TABLE orders USING INDEX idx_user_id (user_id=?)

-- 差的结果（全表扫描）:
-- SCAN TABLE orders
```

### 实战层

**什么时候加索引：**
- WHERE 条件中频繁出现的列
- JOIN 的关联列
- ORDER BY / GROUP BY 的列
- 外键列（Room @ForeignKey 不自动创建索引）

**什么时候不加索引：**
- 表数据量很小（< 1000 行）
- 列的选择性很低（如 boolean 列只有 true/false）
- 频繁写入的表（索引增加写入开销）
- 很少被查询的列

**Room 中分析查询性能：**
```kotlin
// 开启 Room 查询日志
Room.databaseBuilder(...)
    .setQueryCallback({ sqlQuery, bindArgs ->
        Log.d("RoomQuery", "SQL: $sqlQuery, Args: $bindArgs")
    }, Executors.newSingleThreadExecutor())
    .build()

// 使用 Android Studio Database Inspector 查看 EXPLAIN QUERY PLAN
```

**覆盖索引（Covering Index）：**
```sql
-- 如果查询的列都在索引中，不需要回表
CREATE INDEX idx_cover ON orders(status, created_at, amount);
SELECT status, created_at, amount FROM orders WHERE status = 'paid';
-- 直接从索引返回数据，不访问主表
```

## 记忆锚点

索引 = B-Tree 有序结构，查询从 O(N) 降到 O(logN)。复合索引记住最左前缀原则：等值列在前，范围列在后。用 EXPLAIN QUERY PLAN 验证。
