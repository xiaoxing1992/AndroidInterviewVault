# 协程调度器原理

## 问题

> Kotlin 协程的 Dispatchers.Main、IO、Default 底层是如何实现的？它们的线程池策略有什么区别？如何自定义调度器？

## 核心答案

Dispatchers.Default 使用共享的 CPU 密集型线程池（核心数个线程）；Dispatchers.IO 与 Default 共享线程但允许更多并发（默认 64 个）；Dispatchers.Main 在 Android 上通过 Handler(Looper.getMainLooper()) 将任务 post 到主线程。调度器本质是 ContinuationInterceptor，在协程恢复时拦截 Continuation 并切换到目标线程执行。

## 深入解析

### 原理层

**调度器继承体系：**
```
CoroutineContext.Element
  └── ContinuationInterceptor
        └── CoroutineDispatcher
              ├── Dispatchers.Default (DefaultScheduler)
              ├── Dispatchers.IO (DefaultScheduler.IO)
              ├── Dispatchers.Main (HandlerDispatcher)
              └── Dispatchers.Unconfined
```

**核心方法：**
```kotlin
abstract class CoroutineDispatcher : ContinuationInterceptor {
    // 判断是否需要调度（已在目标线程则不需要）
    open fun isDispatchNeeded(context: CoroutineContext): Boolean = true
    // 执行调度
    abstract fun dispatch(context: CoroutineContext, block: Runnable)
}
```

**Dispatchers.Default 实现：**
- 底层是 `CoroutineScheduler`（协程专用的 work-stealing 调度器）
- 核心线程数 = `max(2, CPU核心数)`
- 适合 CPU 密集型任务（计算、排序、JSON 解析）

**Dispatchers.IO 实现：**
- 与 Default 共享同一个 `CoroutineScheduler` 实例
- 但允许的并发数更高：`max(64, CPU核心数)`
- 通过 `LimitedDispatcher` 限制并发，超出的任务排队
- 适合阻塞 IO 操作（网络请求、文件读写、数据库）

**Dispatchers.Main（Android）：**
```kotlin
// AndroidDispatcherFactory 通过 ServiceLoader 加载
internal class HandlerContext(
    private val handler: Handler
) : HandlerDispatcher() {
    override fun dispatch(context: CoroutineContext, block: Runnable) {
        handler.post(block)  // 本质就是 Handler.post()
    }
}
```

**Dispatchers.Unconfined：**
- `isDispatchNeeded()` 返回 false
- 在当前线程立即执行，挂起后在恢复它的线程继续
- 不做任何线程切换

**withContext 切换原理：**
```kotlin
withContext(Dispatchers.IO) { ... }
// 1. 挂起当前协程
// 2. 通过 IO dispatcher 的 dispatch() 将后续代码提交到 IO 线程池
// 3. 执行完毕后，通过原 dispatcher 的 dispatch() 切回原线程
```

### 实战层

**线程池共享的意义：**
- Default 和 IO 共享线程避免线程爆炸
- 一个线程可以同时服务 CPU 任务和 IO 任务
- `limitedParallelism()` 可以从现有调度器创建受限视图

```kotlin
// 自定义限制并发数的调度器
val dbDispatcher = Dispatchers.IO.limitedParallelism(4)
// 最多 4 个协程同时访问数据库

// 完全自定义调度器
val singleThread = newSingleThreadContext("MyThread")
val fixedPool = newFixedThreadPoolContext(4, "MyPool")
// 注意：需要手动 close() 释放
```

**Android 最佳实践：**
- UI 操作 → `Dispatchers.Main`
- 网络/数据库 → `Dispatchers.IO`
- 计算/排序 → `Dispatchers.Default`
- `Dispatchers.Main.immediate` — 如果已在主线程则不 dispatch，直接执行

**坑点：**
- 不要在 `Dispatchers.Main` 上做耗时操作（ANR）
- `Dispatchers.IO` 的 64 并发限制可能不够，高并发场景用 `limitedParallelism`
- 测试时用 `Dispatchers.setMain(testDispatcher)` 替换主线程调度器

## 记忆锚点

调度器 = 拦截器 + 线程池。Default 少线程（CPU 核数）算得快，IO 多线程（64）等得起，Main 就是 Handler.post()。
