# 死锁检测与避免

## 问题

"死锁的四个必要条件是什么？Android 开发中常见的死锁场景有哪些？如何检测和避免死锁？协程中会出现死锁吗？"

## 核心答案（30秒版本）

死锁四个必要条件：互斥、持有并等待、不可抢占、循环等待。Android 常见死锁：主线程 synchronized 等待子线程（子线程又 post 到主线程）、多锁顺序不一致、线程池任务互相依赖。检测工具：ANR trace、StrictMode、jstack、Android Studio Profiler。避免策略：固定加锁顺序、超时获取锁、减小锁粒度、使用无锁数据结构。协程中 Mutex 不可重入也会死锁，Dispatchers.Main.immediate + runBlocking 也是经典死锁。

## 深入解析

### 原理层（源码级）

**死锁四个必要条件（Coffman 条件）：**

```
1. 互斥（Mutual Exclusion）：资源同一时刻只能被一个线程持有
2. 持有并等待（Hold and Wait）：持有资源的线程可以请求新资源
3. 不可抢占（No Preemption）：已持有的资源不能被强制释放
4. 循环等待（Circular Wait）：存在线程等待环 T1→T2→...→Tn→T1
```

**经典死锁模型：**

```java
// 哲学家就餐问题的 Android 版
Object lockA = new Object();
Object lockB = new Object();

// 线程1
synchronized(lockA) {       // 持有 A
    Thread.sleep(100);
    synchronized(lockB) {   // 等待 B → 死锁！
        doWork();
    }
}

// 线程2
synchronized(lockB) {       // 持有 B
    Thread.sleep(100);
    synchronized(lockA) {   // 等待 A → 死锁！
        doWork();
    }
}
```

**JVM 死锁检测原理：**

```
JVM 内部维护一个等待图（Wait-for Graph）：
- 节点 = 线程
- 边 = 等待关系（T1 等待 T2 持有的锁）
- 检测环 = 死锁
```

### 实战层（项目经验）

**Android 常见死锁场景：**

**场景 1：主线程 + 子线程互锁**

```kotlin
// ❌ 经典 ANR 死锁
class DataManager {
    private val lock = Any()
    private var data: String = ""

    // 子线程调用
    fun updateData(newData: String) {
        synchronized(lock) {                    // 子线程持有 lock
            data = newData
            // 需要通知 UI 更新
            Handler(Looper.getMainLooper()).post {
                synchronized(lock) {            // 主线程等待 lock → 死锁！
                    updateUI(data)
                }
            }
            Thread.sleep(1000)  // 模拟耗时
        }
    }

    // 主线程调用
    fun getData(): String {
        synchronized(lock) {                    // 主线程持有 lock
            return data
        }
    }
}

// ✅ 修复：缩小锁范围 + 复制数据
fun updateData(newData: String) {
    val snapshot: String
    synchronized(lock) {
        data = newData
        snapshot = data  // 在锁内复制
    }
    // 锁外 post
    Handler(Looper.getMainLooper()).post {
        updateUI(snapshot)  // 不需要再加锁
    }
}
```

**场景 2：runBlocking 在主线程死锁**

```kotlin
// ❌ 主线程 runBlocking + Dispatchers.Main
class MyActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        // 主线程被 runBlocking 阻塞
        val result = runBlocking {
            withContext(Dispatchers.Main) {  // 等待主线程 → 死锁！
                "data"
            }
        }
    }
}

// ✅ 修复：使用 lifecycleScope
class MyActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        lifecycleScope.launch {
            val result = withContext(Dispatchers.IO) { "data" }
            updateUI(result)  // 自动回到主线程
        }
    }
}
```

**场景 3：线程池任务互相依赖**

```kotlin
// ❌ 单线程池中任务 A 等待任务 B
val executor = Executors.newSingleThreadExecutor()

executor.submit {
    // 任务 A
    val future = executor.submit(Callable {
        // 任务 B — 但线程池只有一个线程，被 A 占用
        "result"
    })
    future.get()  // 永远等不到 → 死锁！
}

// ✅ 修复：使用不同线程池，或增大线程池容量
val executor = Executors.newFixedThreadPool(2)
```

**场景 4：协程 Mutex 不可重入死锁**

```kotlin
// ❌ Mutex 不可重入
val mutex = Mutex()

suspend fun outer() {
    mutex.withLock {
        inner()  // 同一协程再次获取 → 永远挂起 → 死锁
    }
}

suspend fun inner() {
    mutex.withLock {
        // ...
    }
}

// ✅ 修复方案1：重构避免嵌套
suspend fun outer() {
    val data = mutex.withLock { prepareData() }
    processData(data)  // 不在锁内调用 inner
}

// ✅ 修复方案2：使用 owner 检查
val mutex = Mutex()
suspend fun reentrantLock(owner: Any, block: suspend () -> Unit) {
    if (mutex.holdsLock(owner)) {
        block()
    } else {
        mutex.withLock(owner) { block() }
    }
}
```

**场景 5：LiveData + 主线程阻塞**

```kotlin
// ❌ 在子线程 observe LiveData 并阻塞等待
fun getValueBlocking(): String {
    val latch = CountDownLatch(1)
    var result: String? = null

    // observe 必须在主线程
    Handler(Looper.getMainLooper()).post {
        liveData.observeForever { value ->
            result = value
            latch.countDown()
        }
    }

    // 如果当前就是主线程 → 死锁！
    latch.await()  // 阻塞主线程，但 observe 回调也需要主线程
    return result!!
}

// ✅ 修复：使用协程
suspend fun getValue(): String = suspendCancellableCoroutine { cont ->
    val observer = object : Observer<String> {
        override fun onChanged(value: String) {
            liveData.removeObserver(this)
            cont.resume(value)
        }
    }
    withContext(Dispatchers.Main) {
        liveData.observeForever(observer)
    }
}
```

**场景 6：数据库事务嵌套**

```kotlin
// ❌ Room 事务中调用另一个事务方法
@Dao
interface UserDao {
    @Transaction
    suspend fun updateUserAndOrders(user: User, orders: List<Order>) {
```

**死锁检测工具：**

```kotlin
// 1. StrictMode 检测主线程阻塞
StrictMode.setThreadPolicy(
    StrictMode.ThreadPolicy.Builder()
        .detectAll()
        .penaltyLog()
        .penaltyDeath()  // 开发环境直接 crash
        .build()
)

// 2. ANR-WatchDog（第三方库）
class ANRWatchDog(private val timeoutMillis: Long = 5000) : Thread() {
    private val mainHandler = Handler(Looper.getMainLooper())
    private var tick = 0
    private var reported = false

    override fun run() {
        while (!isInterrupted) {
            tick = 0
            mainHandler.post { tick = 1 }  // 主线程执行
            Thread.sleep(timeoutMillis)
            if (tick == 0 && !reported) {
                // 主线程 5 秒未响应
                val stackTrace = Looper.getMainLooper().thread.stackTrace
                reportANR(stackTrace)
                reported = true
            } else {
                reported = false
            }
        }
    }
}

// 3. 自定义死锁检测器
class DeadlockDetector {
    private val lockGraph = ConcurrentHashMap<Long, MutableSet<Any>>()  // 线程 → 持有的锁
    private val waitingFor = ConcurrentHashMap<Long, Any>()             // 线程 → 等待的锁

    fun beforeLock(lock: Any) {
        val tid = Thread.currentThread().id
        waitingFor[tid] = lock
        if (detectCycle(tid)) {
            throw DeadlockException("Deadlock detected! Thread $tid")
        }
    }

    fun afterLock(lock: Any) {
        val tid = Thread.currentThread().id
        waitingFor.remove(tid)
        lockGraph.getOrPut(tid) { mutableSetOf() }.add(lock)
    }

    private fun detectCycle(startTid: Long): Boolean {
        val visited = mutableSetOf<Long>()
        var current = startTid
        while (current !in visited) {
            visited.add(current)
            val waitLock = waitingFor[current] ?: return false
            // 找到持有 waitLock 的线程
            current = lockGraph.entries
                .find { waitLock in it.value }?.key ?: return false
        }
        return current == startTid  // 回到起点 = 环 = 死锁
    }
}
```

**避免死锁的策略：**

```kotlin
// 策略1：固定加锁顺序（破坏循环等待）
class TransferService {
    fun transfer(from: Account, to: Account, amount: Long) {
        // 按 ID 排序，保证所有线程加锁顺序一致
        val (first, second) = if (from.id < to.id) from to to else to to from
        synchronized(first) {
            synchronized(second) {
                from.balance -= amount
                to.balance += amount
            }
        }
    }
}

// 策略2：超时获取锁（破坏持有并等待）
fun safeOperation() {
    val lock1 = ReentrantLock()
    val lock2 = ReentrantLock()

    if (lock1.tryLock(100, TimeUnit.MILLISECONDS)) {
        try {
            if (lock2.tryLock(100, TimeUnit.MILLISECONDS)) {
                try {
                    doWork()
                } finally {
                    lock2.unlock()
                }
            } else {
                // 获取 lock2 超时，释放 lock1 避免死锁
                Log.w("Lock", "Failed to acquire lock2, backing off")
            }
        } finally {
            lock1.unlock()
        }
    }
}

// 策略3：减小锁粒度（减少持有时间）
class UserManager {
    // ❌ 粗粒度：一把大锁
    // private val lock = Any()

    // ✅ 细粒度：分段锁
    private val locks = Array(16) { Any() }
    private fun lockFor(userId: String) = locks[userId.hashCode().and(15)]

    fun updateUser(userId: String, update: (User) -> User) {
        synchronized(lockFor(userId)) {
            // 只锁定特定用户的操作
        }
    }
}

// 策略4：使用无锁数据结构
// ConcurrentHashMap、CopyOnWriteArrayList、AtomicReference
// 协程中使用 Channel 代替共享可变状态

// 策略5：协程中使用 withTimeout 防止无限等待
suspend fun safeWithMutex(mutex: Mutex) {
    withTimeout(5000) {
        mutex.withLock {
            doWork()
        }
    }
}
```

## 记忆锚点

死锁就像"十字路口四辆车互不让路"——每辆车都占着自己的车道（互斥），等对方先走（循环等待）。解决方案：交通规则（固定顺序）、红绿灯超时（tryLock）、修立交桥（无锁设计）。
