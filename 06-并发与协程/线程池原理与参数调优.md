# 线程池原理与参数调优

## 问题

"ThreadPoolExecutor 的核心参数有哪些？任务提交流程是怎样的？拒绝策略有哪些？Android 中的 AsyncTask、协程 Dispatchers 底层是如何使用线程池的？怎么针对 CPU 密集型和 IO 密集型任务调优？"

## 核心答案（30秒版本）

ThreadPoolExecutor 的七大参数：corePoolSize、maximumPoolSize、keepAliveTime、unit、workQueue、threadFactory、handler。任务提交流程：核心线程未满创建核心线程 → 队列未满入队 → 队列满且未达 max 创建非核心线程 → 否则触发拒绝策略。CPU 密集型 corePoolSize = N+1，IO 密集型 = 2N（N 为 CPU 核数）。Android 中 Dispatchers.Default 是 max(2, N-1) 的固定线程池，Dispatchers.IO 默认 64 线程。

## 深入解析

### 原理层（源码级)

**ThreadPoolExecutor 核心参数详解：**

```java
public ThreadPoolExecutor(
    int corePoolSize,              // 核心线程数：常驻线程
    int maximumPoolSize,           // 最大线程数：核心 + 临时
    long keepAliveTime,            // 临时线程空闲存活时间
    TimeUnit unit,
    BlockingQueue<Runnable> workQueue,  // 任务队列
    ThreadFactory threadFactory,   // 线程工厂（命名、优先级）
    RejectedExecutionHandler handler   // 拒绝策略
)
```

**任务提交完整流程（execute 源码）：**

```java
public void execute(Runnable command) {
    int c = ctl.get();
    // 1. 当前线程数 < corePoolSize，创建核心线程
    if (workerCountOf(c) < corePoolSize) {
        if (addWorker(command, true)) return;
        c = ctl.get();
    }
    // 2. 核心线程满，尝试入队
    if (isRunning(c) && workQueue.offer(command)) {
        int recheck = ctl.get();
        if (!isRunning(recheck) && remove(command))
            reject(command);
        else if (workerCountOf(recheck) == 0)
            addWorker(null, false);  // 防止队列有任务但无线程
    }
    // 3. 入队失败（队列满），创建非核心线程到 max
    else if (!addWorker(command, false))
        reject(command);  // 4. 创建失败，拒绝策略
}
```

**四种拒绝策略：**

```java
// AbortPolicy（默认）：抛 RejectedExecutionException
new ThreadPoolExecutor.AbortPolicy()

// CallerRunsPolicy：调用者线程执行（降级处理，不丢任务但阻塞调用者）
new ThreadPoolExecutor.CallerRunsPolicy()

// DiscardPolicy：静默丢弃
new ThreadPoolExecutor.DiscardPolicy()

// DiscardOldestPolicy：丢弃队列最旧任务，重新提交
new ThreadPoolExecutor.DiscardOldestPolicy()
```

**几种常见队列：**

```java
// ArrayBlockingQueue：有界数组队列，FIFO
new ArrayBlockingQueue<>(100)

// LinkedBlockingQueue：默认无界（Integer.MAX_VALUE），危险！
new LinkedBlockingQueue<>(100)  // 务必指定容量

// SynchronousQueue：不存储，必须直接交付
// CachedThreadPool 使用此队列，可能创建无限线程
new SynchronousQueue<>()

// PriorityBlockingQueue：优先级队列，需 Comparable
new PriorityBlockingQueue<>()
```

**Executors 工厂方法的陷阱：**

```java
// FixedThreadPool：核心 = 最大，使用无界队列
// 风险：任务堆积导致 OOM
Executors.newFixedThreadPool(n)
// = new ThreadPoolExecutor(n, n, 0, MS, new LinkedBlockingQueue<>())

// CachedThreadPool：核心 0，最大 Integer.MAX_VALUE
// 风险：无限创建线程导致 OOM
Executors.newCachedThreadPool()
// = new ThreadPoolExecutor(0, MAX_VALUE, 60, S, new SynchronousQueue<>())

// SingleThreadExecutor：单线程，无界队列
// 风险：任务堆积
Executors.newSingleThreadExecutor()

// 阿里巴巴规范：禁止使用 Executors，必须手动创建 ThreadPoolExecutor
```

**Android 中的线程池：**

```kotlin
// Dispatchers.Default 源码（DefaultScheduler）
internal object DefaultScheduler : SchedulerCoroutineDispatcher(
    CORE_POOL_SIZE,        // = max(2, AVAILABLE_PROCESSORS - 1)
    MAX_POOL_SIZE,         // = 1024（理论上限）
    IDLE_WORKER_KEEP_ALIVE_NS,
    DEFAULT_SCHEDULER_NAME
)

// Dispatchers.IO 源码
// 共享 Default 的线程池，但有独立的并发限制（默认 64）
val IO: CoroutineDispatcher = DefaultScheduler.blocking(
    systemProp(IO_PARALLELISM_PROPERTY_NAME,
               64.coerceAtLeast(AVAILABLE_PROCESSORS))
)

// 关键：IO 和 Default 共享线程，避免线程过度切换
// 但 IO 任务标记为 PROBABLY_BLOCKING，调度器会创建额外线程
```

### 实战层（项目经验）

**调优公式：**

```
CPU 密集型：corePoolSize = N + 1
   原因：N 个线程跑满 CPU，+1 防止偶发 IO 阻塞导致 CPU 闲置

IO 密集型：corePoolSize = 2N
   或更精确：N * (1 + W/C)
   W = 等待时间（IO 时间），C = 计算时间
   例如 IO 占 80%：N * (1 + 4) = 5N

混合型：拆分两个线程池，避免相互影响
```

**实战配置模板：**

```kotlin
object ThreadPools {
    private val cpuCount = Runtime.getRuntime().availableProcessors()

    // CPU 密集型：图片处理、加密、数据计算
    val cpuExecutor: ExecutorService = ThreadPoolExecutor(
        cpuCount + 1,
        cpuCount + 1,
        0L, TimeUnit.MILLISECONDS,
        LinkedBlockingQueue(200),  // 有界！
        NamedThreadFactory("cpu-pool"),
        ThreadPoolExecutor.CallerRunsPolicy()  // 降级到调用线程
    )

    // IO 密集型：网络、文件、数据库
    val ioExecutor: ExecutorService = ThreadPoolExecutor(
        cpuCount * 2,
        cpuCount * 4,
        60L, TimeUnit.SECONDS,
        LinkedBlockingQueue(500),
        NamedThreadFactory("io-pool"),
        ThreadPoolExecutor.AbortPolicy()
    )

    // 单线程串行池：日志写入、SP 提交
    val serialExecutor: ExecutorService = Executors.newSingleThreadExecutor(
        NamedThreadFactory("serial-pool")
    )
}

class NamedThreadFactory(private val prefix: String) : ThreadFactory {
    private val counter = AtomicInteger(0)
    override fun newThread(r: Runnable): Thread =
        Thread(r, "$prefix-${counter.incrementAndGet()}").apply {
            priority = Thread.NORM_PRIORITY
            isDaemon = false
            uncaughtExceptionHandler = Thread.UncaughtExceptionHandler { _, e ->
                CrashReporter.report(e)
            }
        }
}
```

**监控线程池健康度：**

```kotlin
class MonitoredThreadPoolExecutor(...) : ThreadPoolExecutor(...) {
    private val taskStartTimes = ConcurrentHashMap<Runnable, Long>()

    override fun beforeExecute(t: Thread, r: Runnable) {
        taskStartTimes[r] = SystemClock.elapsedRealtime()
    }

    override fun afterExecute(r: Runnable, t: Throwable?) {
        val duration = SystemClock.elapsedRealtime() - (taskStartTimes.remove(r) ?: 0)
        if (duration > 1000) {
            Log.w("ThreadPool", "Slow task: $r took ${duration}ms")
        }
        // 监控指标
        Stats.record("pool.size", poolSize)
        Stats.record("pool.active", activeCount)
        Stats.record("pool.queue", queue.size)
        Stats.record("pool.completed", completedTaskCount)
    }
}
```

**踩坑案例：**

1. **OOM 案例**：使用 `LinkedBlockingQueue` 不设容量，任务积压几十万导致 OOM
   - 修复：必须指定队列容量 + 合理拒绝策略

2. **任务饥饿案例**：核心线程都在等待依赖任务（任务 A 提交任务 B 并 await）
   - 修复：拆分线程池，避免同一池内任务依赖

3. **线程泄漏案例**：未捕获异常导致 Worker 线程死亡，线程池逐渐空转
   - 修复：每个任务 try-catch，设置 UncaughtExceptionHandler

4. **优先级反转**：低优先级任务持有锁，高优先级任务等待
   - 修复：使用 `PriorityBlockingQueue` + 任务优先级标识

## 记忆锚点

线程池就像"公司招聘"——核心员工常驻（corePoolSize），订单进队列（workQueue），订单爆满临时招聘（maximumPoolSize），临时工闲置裁员（keepAliveTime），全员爆满启动应急方案（拒绝策略）。
