# 模块化依赖管理策略

## 问题

> 大型 Android 项目如何管理模块间的依赖关系？api 和 implementation 的区别是什么？如何避免循环依赖？如何实现模块的独立编译运行？

## 核心答案

模块化依赖管理的核心是分层（app → feature → domain → data → common）+ 依赖方向单一（上层依赖下层，禁止反向）。api 会传递依赖给消费者，implementation 不传递（编译隔离）。避免循环依赖靠抽取公共接口到独立模块或使用依赖反转。独立编译通过 application/library 动态切换实现。

## 深入解析

### 原理层

**标准分层架构：**
```
app (壳模块，组装所有 feature)
 ├── feature-home
 ├── feature-user
 ├── feature-order
 │     ↓ (依赖方向)
 ├── domain (业务接口/用例)
 │     ↓
 ├── data (Repository 实现、网络、数据库)
 │     ↓
 └── common (工具类、基础组件、UI 组件)
```

**api vs implementation：**
```kotlin
// moduleA/build.gradle.kts
dependencies {
    api(project(":lib-network"))         // 传递：依赖 moduleA 的模块也能用 lib-network
    implementation(project(":lib-utils")) // 不传递：只有 moduleA 内部能用
}

// 影响：
// app → moduleA → lib-network (api)
// app 可以直接使用 lib-network 的类 ✅

// app → moduleA → lib-utils (implementation)  
// app 不能直接使用 lib-utils 的类 ❌ (编译报错)
```

**避免循环依赖的方法：**
```
问题: feature-order 需要调用 feature-user 的用户信息
      feature-user 需要调用 feature-order 的订单数量

方案1: 接口下沉
  domain 模块定义 IUserService / IOrderService 接口
  feature-user 实现 IUserService
  feature-order 实现 IOrderService
  通过 DI 或 ServiceLoader 获取实现

方案2: 事件总线/路由
  feature-order 通过路由获取用户信息
  feature-user 通过路由获取订单数量
  两者不直接依赖

方案3: 中间模块
  抽取 shared-contract 模块放公共数据类
  两个 feature 都依赖 shared-contract
```

**依赖版本统一管理：**
```kotlin
// gradle/libs.versions.toml (Version Catalog)
[versions]
kotlin = "1.9.22"
compose-bom = "2024.02.00"

[libraries]
kotlin-stdlib = { module = "org.jetbrains.kotlin:kotlin-stdlib", version.ref = "kotlin" }
compose-bom = { module = "androidx.compose:compose-bom", version.ref = "compose-bom" }

[bundles]
compose = ["compose-ui", "compose-material3", "compose-tooling"]
```

### 实战层

**独立编译运行：**
```kotlin
// feature-home/build.gradle.kts
if (project.hasProperty("runAlone")) {
    apply(plugin = "com.android.application")
} else {
    apply(plugin = "com.android.library")
}

// 独立运行时提供 Application 和启动 Activity
// src/runAlone/AndroidManifest.xml (sourceSet 切换)
```

**依赖检查工具：**
```bash
# 查看依赖树
./gradlew :app:dependencies --configuration runtimeClasspath

# 检测循环依赖
./gradlew buildHealth  # dependency-analysis-gradle-plugin
```

**最佳实践：**
- feature 模块之间禁止直接依赖（通过 lint 规则或 Gradle 插件检查）
- 公共 UI 组件放 design-system 模块
- 网络/存储等基础设施放 infrastructure 层
- 每个模块有自己的 README 说明职责和边界
- 使用 Convention Plugin 统一模块配置

```kotlin
// convention-plugins/src/main/kotlin/AndroidFeatureConventionPlugin.kt
class AndroidFeatureConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        with(target) {
            apply(plugin = "com.android.library")
            apply(plugin = "kotlin-android")
            apply(plugin = "dagger.hilt.android.plugin")
            dependencies {
                add("implementation", project(":core:domain"))
                add("implementation", project(":core:ui"))
            }
        }
    }
}
```

## 记忆锚点

模块化 = 分层（上依赖下）+ 隔离（implementation 不传递）+ 接口下沉（解循环依赖）。Version Catalog 统一版本，Convention Plugin 统一配置。
