# 跨进程通信架构设计

## 问题

> Android 中有哪些跨进程通信方式？大型 App 多进程架构如何设计？AIDL、Messenger、ContentProvider 各适合什么场景？

## 核心答案

Android IPC 方式包括：Binder（AIDL/Messenger）、ContentProvider、Socket、文件共享、BroadcastReceiver。大型 App 多进程架构通常将主进程（UI）、推送进程、WebView 进程、后台任务进程分离，通过 AIDL 定义服务接口通信。AIDL 适合高频复杂调用，Messenger 适合简单消息传递，ContentProvider 适合数据共享。

## 深入解析

### 原理层

**IPC 方式对比：**

| 方式 | 底层 | 适用场景 | 复杂度 |
|------|------|----------|--------|
| AIDL | Binder | 高频、复杂接口、多线程 | 高 |
| Messenger | Binder (Handler) | 简单消息、串行处理 | 中 |
| ContentProvider | Binder | 数据 CRUD 共享 | 中 |
| BroadcastReceiver | Binder | 一对多通知 | 低 |
| Socket | TCP/UDP | 跨设备、大数据流 | 高 |
| 文件/SharedPreferences | 文件系统 | 低频、非实时 | 低 |
| Bundle/Intent | Binder | Activity/Service 间传参 | 低 |

**多进程架构设计：**
```
主进程 (:main)
├── UI Activity/Fragment
├── ViewModel/Repository
└── 通过 AIDL 调用其他进程服务

推送进程 (:push)
├── 长连接保活
├── 消息接收
└── 通知展示

WebView 进程 (:webview)
├── 隔离 WebView 内存（OOM 不影响主进程）
└── JS Bridge 通信

后台进程 (:work)
├── 大文件上传/下载
├── 数据同步
└── 日志上报
```

**AIDL 通信架构：**
```kotlin
// 1. 定义 AIDL 接口
// IUserService.aidl
interface IUserService {
    UserInfo getUserInfo(String userId);
    void updateUser(in UserInfo user);
    void registerCallback(IUserCallback callback);
}

// 2. Server 端实现
class UserService : Service() {
    private val binder = object : IUserService.Stub() {
        override fun getUserInfo(userId: String): UserInfo {
            return repository.getUser(userId)
        }
    }
    override fun onBind(intent: Intent): IBinder = binder
}

// 3. Client 端连接
class UserServiceConnection : ServiceConnection {
    var service: IUserService? = null
    
    override fun onServiceConnected(name: ComponentName, binder: IBinder) {
        service = IUserService.Stub.asInterface(binder)
    }
    
    override fun onServiceDisconnected(name: ComponentName) {
        service = null
        // 重连逻辑
    }
}

// 4. 使用
bindService(Intent(this, UserService::class.java), connection, BIND_AUTO_CREATE)
val user = connection.service?.getUserInfo("123")
```

**Messenger（简化版 Binder）：**
```kotlin
// Server
class MessengerService : Service() {
    private val handler = Handler(Looper.getMainLooper()) { msg ->
        when (msg.what) {
            MSG_GET_USER -> {
                val reply = Message.obtain(null, MSG_USER_RESULT)
                reply.data = bundleOf("name" to "张三")
                msg.replyTo.send(reply)  // 回复客户端
            }
        }
        true
    }
    override fun onBind(intent: Intent) = Messenger(handler).binder
}

// Client
val messenger = Messenger(binder)
val msg = Message.obtain(null, MSG_GET_USER)
msg.replyTo = Messenger(clientHandler)  // 设置回复通道
messenger.send(msg)
```

### 实战层

**为什么要多进程：**
- WebView 内存隔离（WebView 进程 OOM 不影响主 App）
- 推送保活（主进程被杀，推送进程可能存活）
- 突破内存限制（每个进程有独立的内存上限）
- 模块隔离（插件化场景）

**多进程的代价：**
- Application 会创建多次（每个进程一次）
- 静态变量不共享
- SharedPreferences 不可靠（多进程并发写入）
- 文件锁/数据库锁需要跨进程方案

**连接管理最佳实践：**
```kotlin
// 连接池 + 自动重连 + 死亡监听
class ServiceConnectionManager {
    fun connect() {
        bindService(...)
    }
    
    // 监听服务端进程死亡
    fun onServiceConnected(binder: IBinder) {
        binder.linkToDeath({
            // 服务端进程死亡，自动重连
            reconnect()
        }, 0)
    }
}
```

## 记忆锚点

多进程 = 隔离（内存/稳定性）+ 保活。通信选型：复杂接口用 AIDL，简单消息用 Messenger，数据共享用 ContentProvider。代价是 Application 多次创建、静态变量不共享。
